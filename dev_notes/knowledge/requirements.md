# 监管报送知识库构建 — 需求与约束整理

## 业务背景

### 我的角色
银行监管科技报送业务负责人，需要利用大模型进行数据测试工作，当前缺乏配套的知识库支撑。

### 业务全貌
- 涉及多个报送体系：1104、EAST、征信、反洗钱、账户报送、客户风险、3702、利率报备
- 每个报送体系由不同的 SA/开发/测试团队负责
- 报送数据全部来自上游业务系统，经 Hive 数仓加工后按监管要求报送
- 数据规模：上万张表、2w+ 任务节点，代码加工层级非常深
- 中间有大量校验逻辑、统计脚本、校验脚本（这部分先忽略，不纳入初期范围）

### 业务复杂度
- 业务首先区分**对公**和**个人**
- 每个大类下有多种产品：
  - 个人：微粒贷、分付、We2000 等
  - 对公：对公存款、对公贷款、福费廷、微业贷、信用证等
- 每种产品可能有不同的贷款种类/子分类
- 业务域包括：存款、贷款、同业、理财、外汇
- 一个业务变动通常会影响多个报送体系
- 不同报送体系的表规模差异大：大的有几千张表，小的几十张，关键的主要是几百张

## 现有材料

| 材料 | 说明 |
|------|------|
| ETL 代码 | Git 管理，包含全部加工逻辑 |
| 血缘库 | 完整链路：任务→脚本→SQL→表→字段，支持多维度查询，支持正向和反向追溯 |
| 需求描述 | 来自需求系统，描述可能不够清晰，但有对应的代码变更内容 |
| 监管发文 | PDF、Word、Excel 等格式，包含报送要求和填报规则 |
| 表结构 | 所有表的 DDL，部分表有零散的中文描述 |
| 产品码表 | 标准化的产品分类维度，可作为业务域骨架的生成依据 |

### 血缘库特点
- 链路完整，从报送表可以一路追溯到最上游的业务源表
- 但加工逻辑复杂、层次多，一张报送表的上游可能涉及上千张表
- 大部分上游表并非核心依赖，只是加工过程中用到的辅助表（维度表、参数表等）
- 区分"核心依赖"和"边缘依赖"主要靠：表类型（业务主表 vs 维度表）+ 变更频率 + 人工经验
- 不同报送表的具体场景和逻辑不同，核心依赖的判断标准也不完全统一

## 知识库的目标用户

- **初期**：先整理公共的、大家都需要的基础知识
- **重点**：优先服务测试人员
- **推广**：好用的话，将方法论推广给所有 SA 和开发同学

## 知识库应满足的要求

### 内容要求

1. **业务知识而非代码** — 从代码中提炼业务描述和加工逻辑，但产出物不涉及代码
2. **知识互相关联** — 各知识点之间有明确的关联关系，形成网络而非孤岛
3. **监管发文 Markdown 化** — 尽量将发文内容完整转为 Markdown，保留原文出处引用；无法用 Markdown 表达的内容单独讨论
4. **需求变更记录** — 每个需求要记录：需求内容（含附件）、对应的代码变更范围、知识库影响分析。需求按报送体系拆分记录
5. **历史需求作为参考** — 新需求到来时，能快速找到类似的历史需求，通过历史需求的影响分析为当前需求提供参考

### 组织要求

1. **双轴检索** — 同时支持自上而下（业务域→报送表）和自下而上（报送表→业务域）两种检索方向
2. **每个报送体系独立一套知识库** — 因为由不同团队负责，需要自治
3. **公共业务域知识共享** — 业务知识是跨报送体系的公共知识，不重复维护，各报送体系引用
4. **加工逻辑以报送表为归属** — 最终的加工逻辑按报送表组织
5. **总览到细节的层次** — 内容多，必须有总览层快速导航，再到细节层深入理解
6. **不冗余** — 代码能直接梳理的逻辑不固化到知识库，血缘工具能实时查的关系不重复记录
7. **渐进式加载** — 知识库更多是指引，检索时先加载概览再按需深入

### 构建过程要求

1. **不是一次性生成** — 整理过程中会有大量疑问，需要与熟悉业务的人讨论确认后再生成
2. **人机协作循环** — 大模型生成初稿 → 人工审阅反馈 → 大模型修正 → 确认沉淀
3. **方法论自适应进化** — 随着整理内容的增加，构建方案本身也要进化优化。模板、提取规则、过滤策略都是初始版本，在实践中持续调整
4. **需求驱动更新为主** — 新需求来时更新涉及的知识文档；代码变更作为补充校验

### 质量原则

1. **奥卡姆剃刀** — 如无必要，勿增实体。公共知识简要概述，仅私有/特定知识详细记录
2. **简要清晰** — 用最少的文字表达完整含义
3. **确定性描述** — 口径逻辑无歧义，避免模糊表述
4. **逻辑清晰** — 加工逻辑的描述要有清晰的因果和步骤关系

## 待深入的问题

以下是在讨论中识别出但尚未充分设计的方面：

1. **知识库生成的具体方法** — 大模型如何从代码/血缘/发文中具体提取知识？提取的 prompt 策略是什么？如何处理代码加工层级深的问题？
2. **大模型上下文管理** — 代码量和表数量巨大，单次上下文放不下，如何分批处理？如何利用大模型的上下文存储机制？
3. **检索使用指南** — 不同角色（SA/开发/测试）拿到一个需求后，具体的检索路径和使用流程是什么？
4. **自我进化机制** — 知识库如何评估自身完整性？哪些知识是缺失的？哪些是过时的？如何自动发现需要更新的文档？
5. **评价体系** — 用什么指标衡量知识库的质量和完整性？如何验证生成的知识是否准确？
6. **校验机制** — 生成的业务描述和加工逻辑如何与实际代码/数据交叉验证？如何发现知识库与实际系统的偏差？
7. **业界实践** — 大模型在知识库自动构建领域有哪些成熟的方法论？RAG、GraphRAG、知识图谱等技术如何融入？

## 偏好与约束

- **格式偏好**：Markdown 文档，方便大模型检索和人工阅读
- **架构偏好**：以三层金字塔（总览→主题→明细）+ 双轴索引为基础，后续不够用时引入知识图谱补充
- **核心依赖判断**：主要按表类型（业务主表 vs 维度表）和变更频率，辅以人工经验。不同报送表的标准可能不同
- **产品分类来源**：有标准产品码表，可自动生成业务域骨架
