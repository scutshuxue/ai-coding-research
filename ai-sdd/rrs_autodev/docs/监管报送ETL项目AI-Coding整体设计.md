> 四维运作视角：谁在哪里做什么

---

## 前言：四维视角

传统的架构文档通常从"组件视角"出发——解释 Agent 是什么、Skill 是什么、Wiki 是什么。这种视角适合理解系统构成，但难以回答一个关键问题：**这套系统到底是怎么运作的？**

本文档采用 **"四维运作视角"**，从"谁在哪里做什么"的角度重新组织整套 AI Coding 体系：

- **维度1 · 开发者本地研发** — 开发者在本地工作站上，借助 Claude Code 的 Agent、Skill、Wiki 等能力完成从需求理解到代码生成的全流程开发工作。
- **维度2 · Agentic Loop（ralph loop)** — 多个 Agent 按"执行→自检→纠错→继续"的自主循环模式运转，贯穿需求、架构、编码、测试、审查、部署六大阶段，实现全程自动化并在关键节点引入人工检查。
- **维度3 · AI Coding 服务端** — 在内网服务器上部署私有化 LLM 与业务 Agent，让业务人员通过飞书等即时通讯工具直接提需求，头脑风暴、查数据、追进度，无需理解技术细节，接入dpms。
- **维度4 · 远程服务端与本地协同** — 通过需求流转、进度同步、记忆共享、知识同步四条通道，将服务端与开发者工作站无缝串联，形成"业务提需→开发交付→验收反馈"的完整闭环。

### 四维视角与五层逻辑架构的映射

四维视角是**运作视角**（How it works），五层架构是**逻辑视角**（What it is）。两者并非替代关系，而是互补：

| 五层逻辑架构 |   维度1 本地研发   | 维度2 Agentic Loop |  维度3 服务端  |   维度4 协同   |
| :-----------: | :-----------------: | :----------------: | :------------: | :------------: |
|   L5 协作层   |     HITL检查点     |   Loop编排+自检   |  业务人员交互  |    双端同步    |
|   L4 记忆层   |    Wiki本地副本    |    文档流转载体    |  Mem0+向量DB  | 记忆/知识共享 |
|   L3 执行层   | 11个Agent+3个Skill |    六阶段主流程    |  3个业务Agent  | 需求→开发联动 |
|   L2 规范层   |  CLAUDE.md+SDD模板  |   结构化文档规范   | AGENTS.md配置 |  统一规范传播  |
| L1 基础设施层 | MCP Client+本地工具 |    Hook触发机制    | MCP Server集群 |    网络通道    |

## 第二章：维度1 — 开发者本地研发

> 开发者工作站上的完整 AI 辅助开发能力

### 2.1 Claude Code 入口与 CLAUDE.md 规范

Claude Code 是开发者的主要工作入口。通过 `.claude/` 目录组织所有 AI Coding 配置：

```
.claude/
├── CLAUDE.md          ← 项目规范锚点（版本、命名、禁止行为、Agent/Skill声明）
├── settings.json      ← Hooks 配置（生命周期事件触发）
├── specs/             ← SDD 规范文档（每个任务一份规格书）
├── skills/            ← Skill 技能库
│   ├── table-dependency-analyzer/   → 表血缘分析
│   ├── table-script-finder/         → 表脚本查找
│   └── wtss-dependency-analyzer/    → WTSS依赖分析
└── agents/            ← Agent 配置
```

**CLAUDE.md 的锚点作用**：CLAUDE.md 是所有 Agent 的行为约束源头。它定义了：

- **项目元数据**：数仓分层标准、表命名规范、字段命名规范
- **开发规则**：禁止行为清单、代码风格要求、提交规范
- **Agent/Skill 声明**：哪些 Agent 可用、哪些 Skill 已就绪、调用方式

每个 Agent 启动时都会加载 CLAUDE.md，确保在统一规范下工作。

**上下文模式切换机制**：Agent 不是独立的服务进程，而是 Claude Code 会话中通过斜杠命令（`/requirement`、`/architecture-main`、`/architecture-sub`、`/coding`）触发的上下文模式切换。每次切换时：

1. 加载对应 Agent 的 prompt 配置和专属知识
2. 卸载上一个 Agent 的上下文，释放窗口空间
3. 通过结构化 Markdown 文档传递前序 Agent 的输出

**Agent间文档传递**：Agent 之间**不共享上下文**，而是通过结构化 Markdown 文档传递信息。这确保了：

- 每个 Agent 的上下文干净、专注
- 信息传递可追溯、可审查
- 人工可以在任何环节介入检查和修改

#### 2.3.4 Skill设计原则

- **借用现有平台能力**：将公司已有的数据平台API封装为Skill，不重复造轮子
- **关注代码搜索不准确的信息**：封装人在做需求时会关注但通过grep/find无法准确获取的平台级信息
- **按需加载，节省上下文**：metadata常驻 → body按需加载 → resources仅在使用时加载
- **可组合性**：多个Skill可组合使用，如"先查依赖→再找脚本→再读代码"

### 2.5 知识库自动更新

知识库不是一次性建设，而是随开发过程持续沉淀的活文档。

**Hook触发机制**：通过 Claude Code 的 Hook 事件，在关键节点自动触发知识沉淀：

```
需求完成后（Hook: TaskCompleted）
    ↓

  · 新发现的业务口径 → business_spec/
  · 新增表的主键/字段说明 → data_assets/表知识/

```

**知识沉淀闭环**：

```
开发过程中积累经验
    ↓
Hook 自动提取 + 人工补充
    ↓
知识库 Git 仓库更新
    ↓
其他开发者 / Agent 获取最新知识
    ↓
下次开发质量提升
    ↓ 循环
```

### 2.6 Mem0长期记忆

Mem0 是跨会话的长期记忆系统，与 Wiki 知识库形成互补：

| 对比维度 | Wiki 知识库                | Mem0 长期记忆                        |
| -------- | -------------------------- | ------------------------------------ |
| 内容性质 | 显性知识：规范、文档、定义 | 隐性经验：偏好、教训、模式           |
| 更新方式 | 人工审查 + Git 管理        | Agent 自动写入                       |
| 典型内容 | "DQC校验规则格式是..."     | "上次留存分析，业务要求环比箭头图标" |
| 访问方式 | 文件读取（本地 Git 副本）  | MCP → Mem0 API（向量检索）          |
| 共享范围 | 所有开发者共享（Git同步）  | 开发者端 + 服务端共享（见维度4）     |

**写入时机**：

- 需求完成后：记录需求、方案、业务反馈
- 遇到问题后：记录故障原因和解决方案
- 业务验收后：记录业务人员偏好和反馈意见

**读取时机**：

- 接到新需求时：检索历史相似需求的处理经验
- 架构设计时：参考类似场景的技术方案
- 代码生成时：复用已验证的代码模式

### 2.8 CI/CD发布流程

CI/CD 是代码从开发到生产的桥梁，分为四个阶段：

```
Pipeline 四阶段：

┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ ① Lint   │→│ ② Test   │→│ ③ Scan   │→│ ④ Deploy  │
│          │  │          │  │          │  │          │
│ SQL语法   │  │ 单元测试  │  │ 代码扫描  │  │ 生产部署  │
│ 格式检查   │  │ 集成测试  │  │ 安全扫描  │  │ 调度注册  │
│ 规范校验   │  │ DQC验证  │  │ 依赖审计  │  │ 灰度发布  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
     ↑                                         ↑
  CP5检查点                                  CP6检查点
  代码审查                                   部署验证
```

**触发方式**：

- `git push` → 自动触发 Lint + Test
- Merge Request → 触发全流程（含 Scan + Deploy 到测试环境）
- Tag 发布 → 触发生产部署

**质量门禁**：每阶段都有明确的通过标准，任何一阶段失败则阻断后续流程。Pipeline 完成后通过 Webhook 通知 AI Coding 服务端，触发飞书进度播报。

### 3.1 Agentic Loop 核心概念

Agentic Loop 不是一个具体组件，而是贯穿维度1（本地研发）和维度3（服务端）的**自主执行范式**。其核心模式是：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agentic Loop 核心模式                          │
│                                                                  │
│     ┌──────────┐                                                │
│     │ 外层编排器 │ ← 斜杠命令触发（/requirement, /coding...）     │
│     │ (Human   │                                                │
│     │  指令)    │                                                │
│     └────┬─────┘                                                │
│          │ 任务分配                                              │
│          ▼                                                       │
│     ┌──────────┐     ┌──────────┐     ┌──────────┐             │
│     │  执  行   │────►│  自  检   │────►│  决  策   │             │
│     │          │     │          │     │          │             │
│     │ Agent    │     │ 格式校验  │     │ 通过？    │             │
│     │ 调用Skill │     │ 依赖检查  │     │ Y→继续   │             │
│     │ 读Wiki   │     │ 规范验证  │     │ N→纠错   │             │
│     │ 生成文档  │     │ 完整性检查 │     │          │             │
│     └──────────┘     └──────────┘     └────┬─────┘             │
│                                             │                    │
│                           ┌─────────────────┤                    │
│                           ▼ N               ▼ Y                  │
│                    ┌──────────┐     ┌──────────────┐            │
│                    │  纠  错   │     │  输出 + 下一步 │            │
│                    │          │     │              │            │
│                    │ 错误分析  │     │ 结构化文档    │            │
│                    │ 自动修复  │     │ → 下一个Agent │            │
│                    │ 重新执行  │     │ 或 → 人工审查 │            │
│                    └──────────┘     └──────────────┘            │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

关键特征：

- **自主性**：Agent 在执行中遇到问题会自动尝试纠错，不需要每步人工干预
- **可观测性**：每步的输入输出都是结构化文档，可追溯、可审查
- **有界性**：通过 HITL 检查点限定自主执行的边界，关键决策必须人工确认

  执行：Script/DDL/WTSS/DQC Agent（并行+串行）
  自检：设计完整性检查（每个Step都有对应设计文档）

  门控：[CP3] 开发者确认所有Step设计无误
       │
       ▼
阶段4: /coding（代码生成）
━━━━━━━━━━━━━━━━━━━━━━━

### 3.3 结构化文档作为Loop信息载体

Agentic Loop 中，**文档即接口**。Agent 之间不共享上下文，而是通过严格格式的 Markdown 文档传递信息：

```
文档流转图：

用户需求
  │
  ▼
[Requirement Analyst] ──→ 需求文档(.md)
  │                       · 需求名称/业务目标
  ▼                       · 核心指标+计算口径
[Architecture Main]  ──→ 主架构文档(.md) + Step列表
  │                       · 数据流设计
  │                       · 表血缘分析结果
  │                       · Step分配方案
  ├──→ [Script Agent]  ──→ 脚本设计文档 ──→ [HQL Coding]   ──→ .hql
  │                                    ──→ [Blanca Coding] ──→ .json
  ├──→ [DDL Agent]     ──→ DDL设计文档  ──→ [DDL Coding]   ──→ .sql
  ├──→ [WTSS Agent]    ──→ WTSS设计文档 ──→ [WTSS Coding]  ──→ .job
  └──→ [DQC Agent]     ──→ DQC设计文档  ──→ [DQC Coding]   ──→ .yml
```

**模板规范**：每类文档都有固定的结构模板，存放在 `.claude/specs/` 目录下，确保文档格式一致、信息完整、可被下游 Agent 准确解析。

### 3.4 Loop内自我纠错机制

Agentic Loop 的每个执行阶段都内置自我纠错能力，在交给人工审查之前尽可能自动修复问题：

| 纠错类型       | 检查内容                         | 自动修复方式                                |
| -------------- | -------------------------------- | ------------------------------------------- |
| SQL语法检查    | HiveQL/SparkSQL 语法是否正确     | 解析错误信息 → 定位语法问题 → 重新生成    |
| JSON格式验证   | Blanca配置JSON是否合法           | 格式化修复 → 缺失字段补全 → 重新验证      |
| 依赖合法性检查 | 引用的表、字段、函数是否真实存在 | 通过Skill查询确认 → 不存在则替换为正确引用 |
| 规范一致性检查 | 命名是否符合项目规范             | 对照CLAUDE.md规则 → 自动重命名             |
| 完整性检查     | 所有Step是否都有对应产出         | 识别缺失项 → 补充生成                      |

**纠错边界**：自动纠错最多重试3次。超过3次仍未通过，则暂停 Loop 并提示开发者介入。

### 3.5 Human-in-the-Loop 六大检查点

Agentic Loop 中设置了六个人工检查点（CP1-CP6），确保关键决策有人工把关：

```
[CP1] 需求确认
  触发：需求分析Agent完成需求文档
  审查人：业务分析师
  决策：需求理解是否正确？（指标定义、数据源、质量要求）
  不通过：重新沟通需求，Agent重新分析

[CP2] 方案审查
  触发：主架构Agent完成架构设计
  审查人：技术负责人
  决策：架构方案是否可行？（数据模型、技术选型、风险评估）
  不通过：重新设计方案

[CP3] 设计确认
  触发：子架构Agent完成详设
  审查人：开发者
  决策：我理解这个设计吗？（逻辑正确性、边界条件、异常处理）
  不通过：要求Agent解释或重新设计

[CP4] 测试验证
  触发：Coding Agent生成代码 + 自动化测试
  审查人：开发者 + Quality检查
  决策：测试是否通过？（覆盖率、数据质量、回归测试）
  不通过：修复问题后重测

[CP5] 代码审查
  触发：Merge Request创建
  审查人：Code Reviewer
  决策：代码质量是否达标？（规范符合性、安全隐患、性能问题）
  不通过：驳回修改

[CP6] 部署验证
  触发：生产部署完成
  审查人：运维 / 开发者
  决策：部署是否成功？（调度配置、监控告警、回滚方案就绪）
  不通过：阻止上线，回滚
```

**决策树**：

```
每个检查点的通用决策流程：

Loop 执行到检查点
    ↓
暂停 Loop，通知审查人
    ↓
审查人查看产出文档
    ├── 通过 → Loop 继续执行下一阶段
    ├── 小修改 → 开发者手动调整 → 重新提交审查
    └── 不通过 → Loop 回退到当前阶段重新执行
```

## 第四章：维度3 — AI Coding 服务端

> 服务端与业务人员交互，打通"最后一公里"

### 4.1 为什么需要服务端

在没有服务端之前，业务需求从提出到实现要经历一条很长的传递链路：

```
业务人员 → 口头/邮件描述需求
    ↓ 信息衰减
产品/数据分析师 翻译成技术语言
    ↓ 等待排期
数据开发工程师 理解并实现
    ↓ 理解偏差
业务人员 验收（发现与预期不符）
    ↓
反复沟通修改...
```

这条链路存在三个结构性问题：

- **信息衰减**：每次转述都会丢失细节
- **等待成本**：排期、沟通、确认占据大量时间
- **验收困难**：业务人员看不懂技术文档，只能等结果出来才能反馈

AI Coding 服务端的定位：**让业务人员通过飞书等日常工具，直接与专业 AI Agent 对话，完成需求描述、方案确认、进度跟踪全流程**，无需任何编程能力。

### 4.2 服务端技术选型

服务端需要一个 AI Agent 运行网关，负责接入飞书消息、路由到对应 Agent、管理 Agent 生命周期。技术选型对比：

| -------------- | :--------------: | :---------: | :-------: |
| Agent 编排能力 |   ✅ 原生支持   |   ✅ 支持   | 🔧 需自建 |
| MCP 工具集成   |   ✅ 原生支持   |   ✅ 支持   | 🔧 需自建 |
| 飞书接入       |    🔧 需封装    |  🔧 需封装  | 🔧 需自建 |
| 多Agent路由    |     ✅ 支持     |   ✅ 支持   | 🔧 需自建 |
| 记忆管理       |     ✅ 支持     |   ✅ 支持   | 🔧 需自建 |
| 部署方式       |   本地/服务器   | 本地/服务器 |   灵活   |

### 5.6 业务验收反馈闭环

验收反馈是 Agentic Loop 的"最后一公里"闭合：

```
开发完成 + 部署上线
    ↓
progress-tracker → 飞书推送数据样本
    ↓
业务人员验收
    ├── ✅ 确认上线
    │       ↓
    │   Mem0 写入正向反馈
    │   （记录业务满意点、数据格式偏好等）
    │       ↓
    │   下次类似需求，Agent 自动参考
    │
    └── ❌ 需要调整（附带描述）
            ↓
        Mem0 写入负向反馈
        （记录问题点、修改意见）
            ↓
        需求重新进入开发队列
        附带业务反馈上下文
            ↓
        开发者修改 → 重新部署 → 重新验收
```

**闭环价值**：每次验收的反馈都沉淀到 Mem0，形成持续改进的飞轮——AI 越来越懂业务人员的偏好，首次验收通过率持续提升。

## 第六章：SDD规范驱动开发（贯穿四维的主线）

### 6.1 SDD理念：先规范后编码

SDD（Specification-Driven Development，规范驱动开发）是贯穿四维架构的方法论主线。核心理念：**在AI写代码之前，先完成需求理解、架构设计、任务分解，形成可追溯的规范链。**

```
SDD 的核心流程：

定义规范（Specification）
    ↓ 明确"做什么"和"怎么做"
设计架构（Architecture）
    ↓ 确定技术方案和数据流
生成代码（Coding）
    ↓ 基于规范和架构自动生成
验证质量（Verification）
    ↓ 对照规范检查产出

关键约束：
· 没有规范文档，不允许直接编码
· 规范文档需人工审查通过后，才进入编码阶段
· 代码生成后，必须对照规范进行一致性检查
```

### 6.2 CLAUDE.md 作为规范执行锚点

CLAUDE.md 是 SDD 理念在 Claude Code 中的物化载体。它的作用不仅是"配置文件"，更是**所有 Agent 共同遵守的项目宪法**：

| 规范类别   | CLAUDE.md 中的内容               | 约束效果                     |
| ---------- | -------------------------------- | ---------------------------- |
| 项目元数据 | 数仓分层标准、版本号、技术栈     | Agent 知道"在哪个框架下工作" |
| 命名规范   | 表名/字段名/脚本名命名规则       | AI生成代码自动符合命名标准   |
| 禁止行为   | 不允许DELETE、不允许修改生产表等 | AI不会生成危险操作           |
| Agent声明  | 可用Agent列表、触发命令          | 明确Agent的调用入口          |
| Skill声明  | 可用Skill列表、调用方式          | Agent知道有哪些工具可用      |

### 6.3 结构化输出规范

SDD 要求每个阶段的输出都有固定的模板格式，存放在 `.claude/specs/` 目录下：

**需求规格书模板**：

- 需求名称 / 业务目标 / 核心指标 + 计算口径
- 分析维度 / 时间范围 / 输出形式
- 验收标准 / 数据源 / 质量要求

**架构设计文档模板**：

- 数据流设计（源表 → 加工逻辑 → 目标表）
- 表血缘分析结果（Skill查询产出）
- Step分配方案（各子Agent的任务分工）
- 修改影响评估（哪些现有表/脚本/调度需要变更）

**代码规范模板**：

- SQL代码头注释（需求编号、作者、修改历史）
- Blanca配置结构（source/transform/sink三段式）
- WTSS作业模板（节点定义、依赖声明、超时配置）
- DQC校验规则模板（Aviator表达式、阈值配置）

## 第七章：安全与合规

### 7.1 三道防线

```
┌──────────────────────────────────────────────────────────────────┐
│  第一道防线: 网络隔离                                              │
│  ├── 全链路运行于公司内网，无需暴露公网端口                         │
│  ├── 私有化LLM：代码/数据不出公司网络                              │
│  ├── 飞书连接：WebSocket出站连接，无需开放入站端口                  │
│  ├── 平台API：内网直连，IP白名单                                   │
│  └── CI/CD：代码仓库内网托管，Pipeline内网运行                     │
├──────────────────────────────────────────────────────────────────┤
│  第二道防线: 访问控制                                              │
│  ├── 开发者：通过API Key认证访问LLM                               │
│  ├── 业务人员：飞书企业账号白名单 + 配对机制                       │
│  ├── 平台API：Token + IP白名单双重认证                            │
│  ├── 数据查询：业务人员只能通过只读接口访问聚合数据                │
│  └── CI/CD：基于角色的权限控制（开发者/审核者/管理员）             │
├──────────────────────────────────────────────────────────────────┤
│  第三道防线: 审计日志                                              │
│  ├── LLM调用：全量请求/响应日志（脱敏存储）                       │
│  ├── AI Coding 服务端：业务操作审计日志                            │
│  ├── Skill调用：平台API访问日志                                   │
│  ├── 代码变更：Git提交记录 + MR审查记录                           │
│  └── CI/CD：Pipeline执行日志 + 部署记录                           │
└──────────────────────────────────────────────────────────────────┘
```

### 7.2 监管合规要点

| 合规要求       | 保障措施                                             |
| -------------- | ---------------------------------------------------- |
| 代码不外泄     | 私有化LLM部署，代码仅在内网流转                      |
| 数据不外泄     | 业务数据不发送到外部API，LLM在内网推理               |
| 操作可追溯     | 全链路审计日志 + Git提交记录 + CI/CD日志             |
| 权限最小化     | 业务人员只读、开发者按需访问、CI/CD角色隔离          |
| 监管数据安全   | 监管报送数据在大数据平台内处理，AI仅辅助编写处理逻辑 |
| 代码质量保障   | CI/CD质量门禁 + HITL六大检查点 + 四层防御体系        |
| AI生成代码管理 | AI生成标记 + 人工审查记录 + 责任归属明确             |
