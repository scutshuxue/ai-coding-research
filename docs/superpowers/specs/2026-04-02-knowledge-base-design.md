# 监管报送知识库自动构建方案

## 设计原则

- **奥卡姆剃刀** — 如无必要，勿增实体。公共知识简要概述，仅私有/特定知识详细记录
- **简要清晰** — 每个知识点用最少的文字表达完整含义
- **确定性描述** — 口径逻辑无歧义，避免"大约""可能"等模糊表述
- **不冗余** — 代码可直接梳理的逻辑不入库，血缘工具可实时查询的关系不固化
- **渐进式构建** — 人机协作循环，非一次性生成

## 问题背景

监管报送业务涉及多个报送体系（1104/EAST/征信/反洗钱/账户/3702/利率报备等），数据来自上游业务系统经 Hive 数仓加工。现有上万张表、2w+ 任务节点，代码加工层级深。

现有材料：ETL 代码（Git）、完整血缘库（任务→脚本→SQL→表→字段）、需求描述及对应代码变更、监管发文（PDF/Word/Excel）、表结构（部分有描述）。

目标：利用大模型从上述材料中自动构建结构化 Markdown 知识库，供 SA/开发/测试使用，重点服务测试人员。

## 知识库总体架构

### 三层金字塔 + 双轴索引

```
第一层：总览层
  → 各报送体系入口、业务域全景、交叉矩阵
  → 用途：快速定位方向

第二层：主题层
  → 报送表知识、业务域知识、监管发文、需求变更记录
  → 用途：理解具体业务逻辑和加工规则

第三层：明细层（不固化为文档）
  → 字段级加工逻辑、SQL 转换规则
  → 用途：通过血缘工具实时查询，按需获取
```

**大模型检索路径：** 总览定位方向 → 主题层获取业务知识 → 必要时调用血缘工具深入明细。

### 双轴检索

- **自上而下（业务域 → 报送表）**：从业务变动出发，找到影响的报送表
- **自下而上（报送表 → 业务域）**：从报送表出发，理解涉及的业务背景

两个方向通过交叉矩阵和双向链接实现。

### 目录设计原则

- **每个报送体系完全自治** — 由不同的 SA/开发/测试独立维护
- **公共业务域知识共享** — `business/` 不重复，各报送体系通过链接引用
- **业务域树形结构按产品码表生成** — 对公/个人 → 业务大类 → 具体产品，深度按实际复杂度伸缩
- **每层 index.md 只做导航** — 列出下级分类和一句话描述，不堆积细节

## 构建方法论

### 三阶段循环

```
阶段一：骨架生成（自动化）
  输入：产品码表 + 血缘库 + 报送表清单
  输出：目录结构 + 空壳文件 + 交叉矩阵初稿
  方式：全自动，无需人工

    ↓

阶段二：单表知识填充（人机协作循环）
  每次聚焦一张报送表：
    Step 1 — 大模型提取初稿
    Step 2 — 人工审阅 + 结构化反馈
    Step 3 — 大模型修正
    Step 4 — 确认沉淀 + 更新关联文档
  循环直到质量达标

    ↓

阶段三：方法论反馈（定期回顾）
  每完成 N 张表后回顾：
    → 模板是否合适
    → 提取规则准确率
    → 过滤策略是否合理
  产出：更新模板/规则/策略，记录 changelog
```

### 阶段一：骨架生成

**输入源与产出：**

| 输入源 | 提取内容 | 产出 |
|--------|---------|------|
| 产品码表 | 对公/个人、业务大类、产品明细 | `business/` 目录结构 |
| 血缘库 | 各报送体系末端报送表清单 | `{system}/reporting/` 文件列表 |
| 血缘库 + 产品码表 | 报送表 ↔ 业务产品关联 | 交叉矩阵初稿 |

产出为"空壳"——有目录、有文件、有基本关联，无业务描述和加工逻辑。

### 阶段二：单表知识填充

**Step 1 — 大模型提取初稿**

对选定的报送表，依次执行：

1. 从血缘库拉取该表的上游依赖链
2. 按过滤策略筛选核心上游表（优先业务主表，过滤维度表/参数表）
3. 从 ETL 代码分析主流程加工逻辑，提炼为自然语言
4. 从监管发文中匹配相关章节
5. 按模板组装初稿

**Step 2 — 人工审阅反馈**

反馈采用结构化标注：

| 标注类型 | 含义 | 作用域 |
|---------|------|--------|
| `[多余]` | 该内容不需要记录 | 当前文档 |
| `[缺失]` | 缺少关键信息 | 当前文档 |
| `[不准确]` | 描述与实际不符 | 当前文档 |
| `[粒度不对]` | 太粗或太细 | 当前文档 |
| `[有歧义]` | 口径描述存在多种理解 | 当前文档 |
| `[模板建议]` | 模板结构需调整 | 方法论 |
| `[规则建议]` | 提取/过滤规则需调整 | 方法论 |

**Step 3 — 大模型修正**

根据反馈修正当前文档。`[模板建议]`和`[规则建议]`类反馈记入方法论待优化清单，不立即修改模板。

**Step 4 — 确认沉淀**

文档确认后：
- 更新该报送表关联的业务域文件（增加反向引用）
- 更新 overview.md 中的状态
- 标记 `status: 已确认`

### 阶段三：方法论反馈

**回顾维度：**

| 维度 | 检查内容 | 调整对象 |
|------|---------|---------|
| 模板结构 | 哪些段落总被大改或跳过 | template-*.md |
| 提取准确率 | 哪类逻辑总是提取错误 | extraction-rules.md |
| 过滤策略 | 核心表遗漏或噪声表混入 | filter-strategy.md |
| 粒度标准 | 不同类型表是否需要不同粒度 | template-*.md |
| 分类体系 | 业务域层级是否需要拆分/合并 | business/ 结构 |

所有变更记录在 `_methodology/changelog.md`，标注版本、变更内容、变更原因。

### 需求驱动的增量更新

新需求到来时的更新流程：

```
新需求 → 匹配报送体系
       → 在 changes/index.md 中检索历史类似需求
       → 参考历史需求的"知识库影响分析"
       → 结合血缘工具分析当前需求影响
       → 创建 changes/REQ-xxx.md（需求内容 + 代码变更 + 影响分析）
       → 更新涉及的报送表/业务域知识文档
```

代码变更作为补充校验：ETL 代码合并后，对比变更涉及的表与知识库记录，提示可能需要更新的文档。

## 血缘过滤策略（初始版本）

用于从完整血缘中筛选"核心依赖"，减少噪声：

| 规则 | 说明 |
|------|------|
| 表类型优先 | 业务主表/事实表 > 维度表 > 参数表/码值表 |
| 维度表默认不列入 | 除非人工标注为关键 |
| 层级距离 | 直接上游 1-2 层优先展示 |
| 变更频率 | 高频变更的表更可能是核心依赖 |

此策略为初始版本，随阶段三回顾持续调整。

## 扩展预留：知识图谱

当 Markdown 双向链接不能满足复杂关联查询需求时（如"存款业务变更影响哪些报送体系的哪些表的哪些字段"），可引入知识图谱作为补充：

- 节点：报送表、业务产品、监管发文、上游表
- 边：依赖、影响、引用
- Markdown 知识库作为主体不变，图谱作为索引增强层
- 具体方案在实际使用中根据需求再设计
