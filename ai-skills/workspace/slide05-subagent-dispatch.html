原理三: 子代理派遣模式 — 四层架构

四层审查架构

Layer 1: Implementer

输入: Task 完整文本 (不是文件路径)

工作: 实现 + TDD + 自审

输出: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED

Layer 2: Spec Reviewer

原则: 怀疑 Implementer 报告, 代码比报告可信

检查: 遗漏需求 / 额外功能 / 误解释

Layer 3: Code Quality Reviewer

触发: Spec 审查通过后

检查: 代码质量 / SOLID / 测试覆盖 / 文件分解

Layer 4: Final Code Reviewer

触发: 所有任务完成后, 跨文件一致性

上下文隔离设计

// Controller 精心构造输入

Implementer 输入:

  • Task 完整文本
  • 场景设定 (位置/依赖)
  • 架构上下文

Spec Reviewer 输入:

  • 完整需求 + 实现代码
  • (无会话历史)

Quality Reviewer 输入:

  • git diff + 计划
  • (无实现思考过程)

核心: 每个子代理 = 新上下文空间

不继承历史 -> 避免 Path Dependency

禁止多子代理并行编辑同一文件
