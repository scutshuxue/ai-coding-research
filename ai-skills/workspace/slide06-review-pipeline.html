原理四: 两阶段审查管线 — 做对事 vs 把事做对

审查流程 (Per Task)

1

Implementer 报告 DONE / DONE_WITH_CONCERNS

|

2

Spec Compliance Review — 验证"做对了事"

怀疑 Implementer 报告, 逐行对比需求 vs 实现

| FAIL -> Implementer 修复 -> 重审 (max 3 次)

3

Code Quality Review — 验证"把事做对"

代码质量 / SOLID / 测试覆盖 / 无过度重构

| FAIL -> Implementer 修复 -> 重审 (max 3 次)

4

Mark Task Complete in TodoWrite

| 所有任务完成后

5

Final Code Reviewer — 整体一致性检查

设计洞察

为什么顺序不可互换?

Spec 违规修复后，代码结构会变化

提前做质量审查 = 审查旧代码 = 浪费

先 Spec 再 Quality 避免双重审查

Max 3 次循环的信号

超过 3 次 = 更深层问题:

1. 计划有问题 (任务定义不清)

2. 子代理能力不足 (需要更强模型)

3. 架构设计有缺陷 (需要重新思考)

此时上报人类，不再盲目重试
